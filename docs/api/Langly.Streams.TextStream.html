Remarks

This streams Rune instead of Char to avoid the misunderstandings and complexities working with UTF-16 Code Units causes. Rune represents a higher level that is closer to what people intend to work with.

This fundamentally operates differently from Stream. While it is designed fairly similar in API, the way it operates, even at that level, differs greatly. There's four major components that make up the text stream:

This TextStream type provides orchestration logic around the other components, tying everything together. Through those, it provides the higher level, text-centric, API.

The actual datastream is provided by StreamBase, which implements the basic streaming API for the daatastream. Supporting additional stream types can be done by extended StreamBase, which greatly simplifies adding new datastreams.

Buffers are provided by IReadBuffer and IWriteBuffer, which buffer the basic streaming from StreamBase, enabling features like Peek(out TElement) and limited ISeekable<TElement, TError> behavior even when the underlying stream is not capable.

Encoding is handled by Codec helpers. These handle the translation between encodings through a consistent API.

Additional orchestration can be added by deriving from this type. However, most applications will not need to do this. Instead, the intended mechanism for supporting additional types is through extension methods.

Encoding

The of this stream.

Declaration
public Encoding Encoding { get; }
Property Value
Type Description
Encoding
Remarks

If the current encoding is not known, this will be Unknown. When this happens, if the encoding has to be determined, it will be done so automatically, and then changed to the determined encoding. This always defaults to UTF8 if another specific encoding was not specified at construction or as a BOM.

Determining the stream encoding can be a complex matter.

If the encoding is specified in the constructor, that is the encoding that will be used for all operations. This is the most efficient, so when the encoding is known, specify it upfront.

If the first operation upon the stream is a write and the encoding has not been specified in the constructor, UTF8 will be used.

If the first operation upon the stream is a read and the encoding has not been specified in the constructor, an attempt will be made to buffer and read the BOM. If a BOM is present, the associated encoding will be used. This includes the UTF-8 psuedo-BOM. If no BOM is present UTF8 will be used.

When the encoding is already known, either because it was specified in the constructor, or a write occurred before a read, an attempt to buffer and read the BOM is still made. If no read was possible this is considered successful and everything continues. If a read was possible and there was no BOM present this is considered successful and everything continues. If a read was possible and there is a BOM present, the encoding indicated by the BOM is compared against the specified encoding, failing if it does not match.

Position

The position within the stream, counted by rune offset.

Declaration
public override nint Position { get; set; }
Property Value
Type Description
nint
Overrides
Stream.Position
Remarks

The base Stream type determines position by Byte offset. TextStream overrides this behavior, exposing the position as Rune offset. This is done to simplify and unify the overall concept. If a stream is UTF-8 encoded, advancing two code points could mean advancing two bytes, eight bytes, or anywhere in between. This issue applies to any variable-width encoding, which the majority of modern encoding schemes are. Furthermore, seeking the stream, which can be done through this property, is remarkably difficult to keep track of with byte offset positioning. Translations are done internally to simplify this.
